# Viz Matrices: A Real-time Audio-Reactive Visualizer

**Viz Matrices** is a high-performance, browser-based music visualizer that transforms sound into a captivating dance of light on the Glyph Matrix. It combines a real-time audio analysis and an auto-thresholding pipeline with a fluid simulation engine to generate two distinct, synchronized visualizations: a "fluid" visualization for Bass and three synchronous pulses for highs.

This tool is designed for Nothing Community, creatives and designers who'd appreciate a small PiP of Glyph Matrix visualization of their music by their Figma/Photoshop window while having fine-tuned control over the audio-reactive algorithm.

---

## Core Algorithm Deep Dive

The visualizer's behavior is governed by a sophisticated, multi-stage pipeline designed for performance and artistic control.

### 1. Audio Processing Pipeline

The foundation is the **Web Audio API**, used to create a node-based graph for real-time analysis.

1.  **Source Input**: An `<audio>` element serves as the source node.
2.  **Frequency Band Splitting**: The source is split into three parallel paths for independent analysis of **Bass**, **Mids**, and **Highs**. This is achieved using `BiquadFilterNode` instances configured as `lowpass` (≤250Hz), `bandpass` (250Hz-4kHz), and `highpass` (≥4kHz).
3.  **Analysis**: Each band is connected to its own `AnalyserNode`. These nodes provide raw time-domain (`getFloatTimeDomainData`) and frequency-domain (`getFloatFrequencyData`) data for each animation frame.

### 2. Event Triggering: Adaptive Thresholding

Instead of static triggers, Viz Matrices uses an adaptive thresholding system to react intelligently to the music's dynamics.

1.  **Peak History (Memory Bank)**: For the Bass and Highs bands, the system maintains a rolling history of the maximum amplitude peaks detected in the time-domain data. The duration of this history is configurable (`memoryBankSeconds`).
2.  **Representative Peak Calculation**: On each frame, the system sorts the peak history and finds a high percentile value (e.g., 90th percentile). This serves as a "representative peak" for the recent dynamics of the track, ignoring occasional quiet moments or extreme outliers.
3.  **Dynamic Threshold Adjustment**: The actual trigger threshold isn't static. It smoothly interpolates towards a target value calculated from the representative peak (`target = representativePeak * sensitivity`). This allows the visualizer to remain sensitive during quiet passages and avoid over-saturation during loud sections.
4.  **Configurable Parameters**:
    *   `sensitivity`: Controls how close to the representative peak the signal must be to trigger an event.
    *   `floor`: A minimum threshold to prevent ambient noise from causing triggers.
    *   `smoothing`: Controls the interpolation speed, preventing jarring threshold changes.

When an incoming amplitude peak surpasses this dynamic threshold, a "hit" event is generated for that frequency band.

### 3. 'Ripple' Visualization Engine

The core of the visualizer is a 2D fluid dynamics simulation based on a simplified wave equation solver.

1.  **Simulation Grid**: The simulation runs on a low-resolution grid (e.g., `128x128`). This ensures high performance.
2.  **Two-Buffer System**: Two float arrays (`currentBuffer`, `previousBuffer`) store the height (amplitude) of the "fluid" at each grid point. The next state of a point is calculated based on the average height of its four neighbors in the `currentBuffer` minus its own height in the `previousBuffer`.
    ```
    // Simplified simulation logic
    newValue = (neighborsSum / 2) - previousValue;
    ```
3.  **Damping & Viscosity**: The `newValue` is multiplied by a `viscosity` factor (e.g., 0.96-0.999). This damping simulates energy loss, causing the waves to dissipate over time.
4.  **Absorption Border**: To prevent waves from reflecting unnaturally off the simulation edges, a stronger damping factor is applied to the grid's border region, effectively absorbing the wave energy.
5.  **Disturbance Mapping**:
    *   **Bass Hits**: Trigger a central disturbance in the simulation. The `dropStrength` and `bassRadius` parameters control the force and area of this impact, creating the main ripple effect. A `rippleCooldown` prevents the simulation from becoming oversaturated.
    *   **Highs Hits**: Activate a pulsing effect on the high-resolution renderer. This is kept separate from the fluid simulation to maintain visual clarity.

### 4. Rendering Pipeline

The final output is generated by two distinct renderers that use the same source data.

1.  **Composite Image Generation (`Visualizer.tsx`)**:
    *   The low-resolution simulation buffer is upscaled and drawn onto a high-resolution, circular canvas.
    *   The fluid data is remapped through post-processing controls (`contrast`, `blackPoint`, `whitePoint`) for artistic effect.
    *   The orbital "Highs" pulses are rendered on top using `globalCompositeOperation = 'lighter'`.
    *   The final state of this canvas is captured as an `ImageData` object on every frame.

2.  **Pixel Matrix Renderer (`PixelMatrix.tsx`)**:
    *   This renderer receives the composite `ImageData` from the high-resolution visualizer.
    *   It downsamples this image to a `25x25` circular glyph matrix.
    *   For each "pixel" in the matrix, it averages the brightness of the corresponding region in the source `ImageData`. This ensures the glyph is a faithful, low-resolution representation of the main visual.
    *   An optional `glow` effect is applied using the canvas context's `shadowBlur` property, with intensity and color derived from the pixel's brightness.

This decoupled rendering pipeline ensures that both visualizers are perfectly synchronized while allowing for independent styling and optimization.
